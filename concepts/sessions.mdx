description: "How Era maintains identity continuity and separates conversations."

## Identity inputs

Era requires `visitorId`, `partnerEndUserId`, or both for each `turnContext()` call.

- `visitorId` maintains continuity for a browser visitor. Obtain it from the [Web SDK](/web-sdk/quickstart).
- `partnerEndUserId` is your stable identifier for an authenticated user. Supply it only from a trusted backend.
- Supplying both connects the browser conversation with the partner user identity.

You can set defaults when you create `Era`:

```typescript
const era = new Era({
  apiKey: "...",
  baseUrl: "...",
  partnerId: "...",
  visitorId: "era_vid_00000000-0000-4000-8000-000000000000",
  sessionId: "session_xyz",
});
```

For a shared server instance, pass identity per call:

```typescript
const block = await era.turnContext(messages, {
  visitorId,
  partnerEndUserId: authenticatedUser.id,
  sessionId,
});
```

If neither continuity identifier is present, the SDK fails open with `""` and does not make a request or invent an identity.

## Trusted identity bridges

When `partnerEndUserId` is present, your trusted backend can also provide an email address or phone number:

```typescript
const block = await era.turnContext(messages, {
  visitorId,
  partnerEndUserId: authenticatedUser.id,
  email: authenticatedUser.verifiedEmail,
  phone: authenticatedUser.phone,
  sessionId,
});
```

Bridge assertions must come from data your backend trusts. Do not forward an email address or phone number directly from browser input without validating ownership.

## Session IDs

Use one stable, unique `sessionId` for the lifetime of a conversation. Create a new value when the user starts a new conversation. Do not reuse the same session across unrelated conversations.

If you omit `sessionId`, the SDK derives one from the first user message with SHA-256. That fallback is deterministic, so two conversations that start with the same text can receive the same derived ID.

<Note>
  For production chat, pass your application's conversation ID as `sessionId`. The derived ID is convenient for simple examples and tests, but it is not a substitute for a unique conversation identifier.
</Note>

## Identity-aware memoization

The TypeScript SDK memoizes the latest turn lookup. Its key includes:

- `visitorId`
- `partnerEndUserId`
- `email` and `phone`
- `sessionId`
- The current one-based `turn_index`

Repeated middleware calls for the same latest turn can share one result. A change to any identity assertion creates a different key, which prevents a cached block from one identity context being reused for another.

```typescript
const options = { visitorId, partnerEndUserId, sessionId };

const block1 = await era.turnContext(messages, options);
const block2 = await era.turnContext(messages, options);

// Duplicate calls for the latest turn share the memoized result.
```

The memoized entry belongs to each `Era` instance. Construct one shared instance for a server process and pass identity and session data per request.

```typescript
const era = new Era({ apiKey, baseUrl, partnerId });

async function handleTurn(
  visitorId: string,
  partnerEndUserId: string | undefined,
  sessionId: string,
  messages: Message[],
) {
  return era.turnContext(messages, {
    visitorId,
    partnerEndUserId,
    sessionId,
  });
}
```
