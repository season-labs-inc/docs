## What is Era?

Era is not memory. Memory systems remember facts about a user — "has a dog named Rex." Era answers a different question: **why is this user here right now, what do they want, how urgent is it, and what should your agent do about it?**

Every turn, Era reads the conversation and returns a typed context block — state, goal, values, intent, and a recommended action — that you inject into your system prompt. One call per turn:

```
[STATE v1] situation=duplicate_charge_on_card urgency=high knowledge=knows_refund_policy [/STATE]
[GOAL v1] active=do:reverse the duplicate charge journey=[seek:find charge origin → do:reverse the duplicate charge] resolved=unresolved [/GOAL]
[VALUE v1] top=[speed:high, trust:moderate] [/VALUE]
[INTENT v1] intent=deciding(0.84) specificity=specific journey=[curious,validating,deciding] [/INTENT]
[CAUSAL v1] trigger=statement_closes_tomorrow prior_attempt=contacted_bank_first emotion=frustrated need=fast resolution need_level=safety [/CAUSAL]
[REC v1] action=validate channel=in_app timing=0 do="Confirm the duplicate and start the refund — skip re-verification." rationale="User has said 'quickly' twice; identity already verified turn 1." priority=1 [/REC]
```

Your agent reads this and stops behaving generically — it knows the user is frustrated, values speed over thoroughness, and needs escalation, before writing a single word.

If you already run a memory layer (Mem0, Zep, or your own), keep it. Memory tells your agent what's true; Era tells it what's happening. The two blocks inject side by side.

## How it fits into your stack

```
User message
     │
     ▼
era.turnContext(messages)   ←── Era service (non-blocking in async mode)
     │
     ▼
Inject block into system prompt
     │
     ▼
LLM call  →  Response to user
```

Era runs **before** your LLM call. In `async` mode (the default) it returns instantly with the previous turn's precomputed block, so there is zero added latency on the critical path.
